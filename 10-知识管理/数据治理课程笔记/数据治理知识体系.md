# 数据治理知识体系

> 按一条数据从"摸清家底"到"安全合规"的完整链路组织。

---

## 全局视角：数据治理做什么？

```
数据盘点（摸家底）
    ↓
数据接入（搬进来）
    ↓
元数据管理（贴标签，让数据可查找）
    ↓
数据仓库（分类存放，记历史账）
    ↓
主数据管理（跨系统同步，一个真相）
    ↓
数据质量（查毛病，保可用）
    ↓
数据安全（守底线，合规不翻车）
    ↓
数据服务（编目→挂载→共享，对外提供）
    ↓（政务版 + 企业版，两条路径）
```

> 狭义的数据治理 = 标准 + 元数据 + 质量 + 安全。广义的数据治理 = 从盘点接入到对外服务的全链路。

## 一、数据盘点（摸家底）

### 一句话

正式做数据接入前，先搞清楚自己家里到底有多少系统、多少表、哪些能用、哪些是垃圾，顺便摸清业务部门真实痛点。

### 三大目的

1. **摸底家底**：多少系统在跑、多少表有价值、数据字典是否完整。很多项目上百个系统上万张表，部分系统已停用仍在被维护，浪费资源。
2. **为后续建模做准备**：提前理清数据结构和业务含义，避免接入后返工。
3. **摸清业务部门真实需求**：甲方频繁变更需求根本原因是不了解他们痛点。盘点中洞察痛点 → 引导需求 → 大幅减少后期变更。

### 执行流程

| 阶段 | 做什么 | 关键产出 |
|---|---|---|
| ① 业务调研 | 画业务流程图，识别数据流转卡点和不合理设计 | 业务→系统→数据的映射关系 |
| ② 系统盘点 | 统计对接方式（数据库直连/文件）、数据库类型（Oracle/MySQL/国产）、数据量级 | 系统盘点清单 |
| ③ 表级分析 | 按**二八原则**只接核心 20% 表，过滤临时表/汇总表/备份表/测试数据/空表 | 表级清单 + 字段含义 + 代码项标注 |

### 企业 vs 政务

| 维度 | 企业 | 政务 |
|---|---|---|
| 协调难度 | 低，高层牵头即可 | 高，几十个委办局 |
| 推进方式 | 开会 + 邮件 | 红头文件 + 专班集中办公 |
| 核心工具 | 调研问卷 + 系统清单 | **三清单**（责任/需求/负面）+ 共享交换平台 |
| 数据获取 | 相对直接 | 必须走共享交换平台，不能私下点对点传 |

### 🔑 实战技巧：没有数据字典怎么办？

**反向模拟法**：让业务人员做一次操作（如新增客户），去后台看哪些表的哪些字段变了，反向推断字段含义。比求甲方翻古董文档快十倍。

## 二、数据接入（搬进来）

### 一句话

把不同类型的数据源汇总到统一的大数据平台，为后续治理和分析提供原材料。

### 数据类型

| 类型 | 典型代表 | 特征 |
|---|---|---|
| 结构化 | MySQL/Oracle 关系库 | 二维表，数据治理主流场景 |
| 半结构化 | Kafka/MongoDB/ES/HBase | JSON 串、对象存储 |
| 非结构化 | 图片、文件、音视频 | 格式不固定 |
| 实时流 | Flink CDC + 传感器/消息队列 | 日志变化捕获或事件触发 |

### 三种接入方式

| 方式 | 适用场景 | 特点 |
|---|---|---|
| 整库接入 | 集群初始化，一次性搬历史数据 | 批量快，可过滤不要的表 |
| 单表接入 | 日常增量，每天按表配置策略 | 灵活，每张表单独处理 |
| SQL 接入 | 特定筛选条件 | 灵活但用得少 |

> 典型流程：新集群→整库初始化历史数据→之后每天单表接入增量。

### 增量策略（做完初始化后每天怎么接）

| 策略 | 做法 | 适用场景 |
|---|---|---|
| 全量覆盖 | 每天拉全量，覆盖前一天 | 代码表/编码表（不存历史） |
| 全量追加 | 每天拉全量，带时间戳追加 | 核心业务表（存历史快照，有冗余） |
| 增量追加 | 只拉新增数据，追加到历史后 | 日志表/流水表（历史不变） |

> **CDC（如 Flink CDC）是实现增量接入的一种技术手段**——监听数据库日志，捕获 insert/update/delete 实时同步。别把它和上面的策略混为一谈。CDC 是实现方式，覆盖/追加是存储策略。

### 常用工具

| 工具 | 类型 | 适用规模 |
|---|---|---|
| Sqoop | Hadoop 开源 | ⭐ 最常用稳定，大批量 OK |
| DataX | 阿里开源 | 单机版，社区不活跃 |
| Kettle | 开源 ETL | 小数据量，大场景不稳定 |
| Informatica / Data Stage | 商用 | 企业级，贵 |

## 三、元数据（贴标签）

### 一句话

元数据 = 描述数据的数据。就像书的标题、作者、ISBN 描述的是书的内容。

### 为什么重要

没有元数据，你看到一堆 `1, 2, 20, 10, 40` 根本不知道这些数字是年龄还是价格。**元数据让数据从"看不懂的数字"变成"可用的信息"。**

### 三大分类

| 分类 | 举例 | 获取方式 |
|---|---|---|
| 技术元数据 | 字段名、表名、数据库类型、JDBC 信息 | 技术手段自动获取 |
| 业务元数据 | 业务术语定义、指标口径、数据所有者 | 需要人工完善 |
| 管理元数据 | 权限、安全等级、归档策略 | 需要人工完善 |

### 元数据 vs 数据元

- **元数据**：描述数据的数据（宽泛概念）
- **数据元**：描述数据的最小单元标准（如国标规定"性别"字段取值 1=男 2=女）

建表建字段时要遵循数据元标准——就像图书馆上架新书必须按国标分类法贴标签。

### 应用场景

- **数据地图 / 数据资产目录**：让使用者能搜索和发现数据
- **影响分析**：改一个字段，追溯下游哪些报表会受影响
- **血缘分析**：知道数据从哪来、经过什么加工、最终去了哪

## 四、数据仓库（分类存放，记历史账）

### 一句话

数据仓库不是某个具体产品（Oracle/MySQL 是产品），而是一整套方法论和建设过程——把多源数据抽取、清洗、转换后，按主题分类存储，记录历史变化，支撑分析决策。

### 数据库 vs 数据仓库

| 维度 | 数据库 | 数据仓库 |
|---|---|---|
| 定位 | 具体产品（Oracle/MySQL/达梦） | 方法论 + 建设过程 |
| 数据特征 | 当前时点，记录被覆盖 | 历史变化，带时间戳 |
| 用途 | OLTP（支撑业务交易） | OLAP（支撑分析决策） |
| 数据范围 | 单一业务系统 | 集成多源多类型 |
| 组织方式 | 按业务流程 | 按主题分类 |

### 四大特性

| 特性 | 解释 |
|---|---|
| ① 面向主题 | 按主题分类（客户/订单/产品）。就像新发地市场——蔬菜大厅、牛羊肉大厅，要什么去对应大厅 |
| ② 集成 | 不同来源数据汇总到统一仓库，就像全国农产品通过铁路飞机汽车运到新发地 |
| ③ 相对稳定 | 入库后不修改，只用于分析。不像业务系统频繁增删改 |
| ④ 反映历史变化 | 保留历史快照。数据库只存当前状态，数仓能告诉你"去年今天这个客户的手机号是多少" |

> ⚠️ 关于"主键"：事实表（Fact Table）通过时间戳+外键组合定位记录，无传统意义的单一主键；但**维度表（Dim Table）有代理主键（Surrogate Key）**。面试别说"数仓所有表都没主键"。

### 分层架构

```
业务系统 → ODS（贴源层，原样接入）
              ↓
           DW（汇总层，清洗转换）
              ↓
           DM（数据集市层，面向具体分析场景）
              ↓
           报表/指标
```

### 没有数仓会怎样？

| 维度 | 直接查业务库 | 有数据仓库 |
|---|---|---|
| 查询速度 | 海量表多表关联，慢 | DW 层取汇总指标，快 |
| 稳定性 | 上游字段/逻辑变化，下游崩 | 做了清洗转换，不受影响 |
| 历史回溯 | 业务系统只存当前 | 历史数据随时查 |

## 五、主数据（跨系统同步，一个真相）

### 一句话

主数据 = 同时存在于两个以上业务系统的共享数据。CRM 改了客户等级，ERP 和财务也得跟着改——主数据管理系统（MDM）就是在中间做统一调度。

### 判断标准

一个数据要叫主数据，须同时满足：① 跨至少两个系统存在 ② 被多部门共同引用 ③ 一个地方变了，其他地方必须同步。

典型：客户、产品、供应商、组织架构、人员（企业）；人口数据（政务——公安、民政、税务、法院等几十个委办局共享）。

### MDM 运作方式

```
业务系统变更 → 通知 MDM → 统一清洗校验 → 同步分发到所有相关系统
```

关键是**双向的**——接收变更，再分发回去。

### 主数据 vs 数据仓库

| 维度 | MDM | 数据仓库 |
|---|---|---|
| 定位 | 业务系统 | 分析系统 |
| 数据流向 | **双向**（收变更+发同步） | **单向**（取数据+出分析） |
| 数据范围 | 跨系统核心共享数据（子集） | 全面数据（全集） |
| 实时性 | **极高**（库存超卖 = 事故） | 相对低（多数 T+1） |

> 经典场景：抖音、淘宝、京东、线下门店共享一份库存。一个渠道卖出一件，所有渠道实时同步——靠的是 MDM，不是数仓。

### MDM 和数仓怎么配合

- 先建 MDM → 核心数据已干净了，数仓省一半力气
- 先建数仓 → 从数仓识别出主数据，同步到 MDM 做初始化
- 两条路互相兜底，不二选一

## 六、数据质量（查毛病，保可用）

### 一句话

用规则自动检查数据加工过程中的异常，发现问题报警，避免影响业务。

### ⚠️ 先说一个坑

数据质量的完整体系是**六个维度**：完整性、一致性、准确性、及时性、唯一性、有效性。目前课程笔记只讲了后两个场景（及时性稽核 + 拉链表稽核），前四个维度还没覆盖。

### A. 及时性稽核

离线数据加工通常凌晨 12:30 开始，第二天上班前必须跑完。**每个环节设时间阈值，超时就报警，不等早上发现再追责。**

| 监控点 | 报警时机 |
|---|---|
| ODS 层某表 ETL_date | 超预期时间未完成 |
| DW 层关键表加工 | 超出时间阈值 |
| 工作流整体结束 | 超正常结束点 |

> 核心原则：报警前置。凌晨发现问题半夜修，不影响第二天业务看数据。

### B. 拉链表稽核

拉链表（SCD Type 2）= 数仓记录历史变化的手段。用 start_date + end_date 两个字段管理记录的生效和关闭。

查询公式：`WHERE start_date <= 查询日期 AND end_date >= 查询日期`

三种异常：

| 异常 | 说明 |
|---|---|
| 断链 | 时间线有空隙，某时间段缺少有效记录 |
| 交叉链 | 同一用户两条记录时间范围重叠 |
| 重复链 | 同一时间存在多条完全相同的记录 |

## 七、数据安全（守底线）

### 一句话

在数据全生命周期里，通过技术+制度防止泄露、篡改、滥用。数据上升为第五大生产要素后，安全从"可选项"变成了"强制项"。

### 法律时间线

| 时间 | 法律/标准 | 要点 |
|---|---|---|
| 2015.07 | 国家安全法 | 奠定网络与信息安全保障框架 |
| 2016.11 通过<br>2017.06 生效 | 网络安全法 | 专门立法规范网络安全 |
| 2019.05 | 等保 2.0（GB/T 22239） | 新增云计算、移动互联网、物联网、工控、大数据安全五大扩展 |
| 2021.06 | 数据安全法 | 里程碑——鼓励数据依法流通，促数字经济发展 |
| 2021.08 | 个人信息保护法 | 规范个人信息收集使用 |

> ⚠️ 等保评估的是**整个组织**——机房、硬件、网络、制度体系、软件全涵盖。你开发的系统只是其中一部分，但要满足密码复杂度、验证频率等软件层面要求。政府项目一般要求过等保三级。

### 两个典型案例

| 案例 | 后果 | 教训 |
|---|---|---|
| Facebook 5.33 亿用户泄露 | 罚 2.65 亿欧元（≈20亿人民币） | 数据安全做不好是灭顶之灾 |
| 山东某公司（国内首起数安法处罚） | 行政警告 | 采集信息放公有云但没加密脱敏→即使没售卖也违法 |

### 数据安全框架（五层）

| 层面 | 内容 |
|---|---|
| ① 原则 | 合法正当、目的明确、最小够用、全程可控、动态控制、权责一致 |
| ② 等级 | 1公开 → 2合作 → 3个人财产 → 4支付信息 → 5国家安全 |
| ③ 全链路防护 | 防火墙、身份认证、加密、脱敏、水印、权限、审计 |
| ④ 组织保障 | 明确负责组织+制度（申请流程、等级设置、权限设置） |
| ⑤ 运维保障 | 访问控制、边界管控、安全测评 |

### 数据加密

| 类型 | 原理 | 算法 | 场景 |
|---|---|---|---|
| 对称加密 | 加解密同一密钥 | DES/3DES/AES | 速度快，密钥分发是难点 |
| 非对称加密 | 公钥加密，私钥解密 | RSA | 安全但慢 |
| 国密算法 | 国家密码局认证 | SM2/SM3/SM4 | 政府、央企**强制要求** |

> 加密是**可逆**的（有密钥就能解开）。需要不可逆保护的用脱敏。

### DSMM（数据安全能力成熟度模型）

2021 年国标，以数据全生命周期为中心，从组织建设、制度流程、技术工具、人员能力四个维度评 1-5 级。

## 八、政务数据服务（编目→挂载→共享）

### 一句话

数据治理做完了，数据资产形成了，怎么对外提供？政务领域通过"编目→挂载→级联→申请→共享交换"这条链路，让数据在合规前提下被找得到、拿得到、用得上。

### 🖼️ 一个比喻：馆际互借系统

把整个政务数据服务体系想象成**全国图书馆馆际互借系统**：

- **编目** = 每本书上架前先登记到书目系统（书名、作者、分类号）。没有编目，读者翻遍书架也找不到
- **挂载** = 书目系统里写了"有这本书"，书架上真得有这本书才行。目录和实体连上
- **上下级级联** = A 市图书馆的书目同步到省图书馆，省图书馆同步到国家图书馆。读者在任何一个图书馆都能搜到全国的书
- **共享网站** = 内部馆际互借系统——只有各图书馆工作人员能用，调书走内部物流
- **开放网站** = 对普通读者的公开查询系统——谁都能搜，但借书要申请
- **有条件共享** = 珍本善本——不是谁都能借，馆长审批通过才能调
- **无条件共享** = 普通藏书——申请即调
- **单次使用** = 借一次，用完归还
- **多次使用** = 定期订阅——这本期刊每期的更新都自动推送过来

> 读者不需要知道书存在哪个城市的哪个书架，只需要在目录里搜索→申请→审批→调书到手。政务数据共享同理：委办局不用知道数据在哪个部门的哪个服务器，目录里找到→申请→审核→交换平台搬到前置机。

### 整体架构：四大板块

| 板块 | 说明 |
|---|---|
| **① 共享网站** | 政务内网部署，各厅局委办局内部共享数据。互联网访问不到 |
| **② 开放网站** | 对公众开放，普通民众可以查看和申请政府发布的公开数据 |
| **③ 资源目录系统** | 数据编目、注册、变更、申请的操作入口——就像图书馆的书目检索系统 |
| **④ 共享交换平台** | 真正的"搬运系统"——审核通过后，把数据从提供方前置机搬到需求方前置机 |

四大板块外围：**标准规范**（编目国标、交换国标）+ **安全保障**（制度和安全体系）。

### 核心流程：四步走

#### 第一步：资源目录编目

把政务数据按国家信息资源编目标准，分类编制成目录。分类维度通常是：**基础资源目录**（人口、法人、电子证照、空间地理、信用基础）、**主题资源目录**、**部门资源目录**。

> 就像图书馆给每本书贴分类标签——读者先翻目录卡，再去找书。

#### 第二步：上下级平台级联

- 省级目录 → 上报到国家
- 市级目录 → 上报到省级
- 部委体系（如政法委）→ 部级下发目录标准 → 各省按标准编目 → 回传部级

**核心目的**：解决跨地域、跨部门的数据资源共享。你想从山西省访问山东省的数据，通过国家级的目录和共享交换平台。

#### 第三步：资源挂载

目录编好了，把目录和数据源连起来——让目录不只是"标题"，而是后面真有数据。挂载完成后发布。

#### 第四步：申请使用

用户在资源目录系统里**检索目录** → 找到需要的数据 → 发起申请 → 审核 → 共享交换平台配置交换任务 → 数据从提供方前置机搬到需求方前置机。

### 两种共享方式

| 方式 | 流程差异 | 适用场景 |
|---|---|---|
| **无条件共享** | 申请后直接配置交换任务，无需审核 | 基础公开数据 |
| **有条件共享** | 申请 → 审核（提供方判断理由是否充分）→ 通过后配置交换 | 敏感/受限数据 |

### 两种使用频率

| 频率 | 做法 | 适用场景 |
|---|---|---|
| **单次使用** | 申请通过后一次性下载数据 | 偶尔需要某份数据 |
| **多次使用** | 配置定期调度任务（如每天凌晨同步增量） | 结构化库表，需要持续获取更新数据 |

> 这四种方式两两组合出不同场景——无条件+单次最简单；有条件+多次最复杂（加审核+配调度）。

### 三种数据获取方式

共享交换平台支持三种方式拿数据：① **库表**（直接拿数据库表）② **接口**（API 调用）③ **文件**（文件传输）。

### 实施流程（按角色分配）

| 环节 | 谁做 | 做什么 |
|---|---|---|
| 目录设计、上下级级联 | 实施咨询人员 | 业务操作，不是技术活 |
| 资源挂载、数据汇聚 | 数据技术人员 | 网络打通、数据接入 |
| 数据共享、更新 | 各委办局人员 | 日常使用共享平台 |
| 数据应用 | 各厅局数据使用人员 / 社会公众 | 查询、申请、使用数据 |

### 与之前知识的关系

- 数据盘点时做的**三清单**（责任/需求/负面）→ 就是编目的前置输入。责任清单决定目录里有什么，负面清单决定什么不能共享
- 数据接入时建的**数据资源池** → 就是挂载阶段要连接的数据源
- 数据安全里的**原则**（目的明确、最小够用、全程可控）→ 就是共享申请审核的依据

## 九、企业数据服务（资产目录 + API + 同步）

### 一句话

企业里数据治理做完了，数据怎么给业务方用？三种途径：**JDBC 直连、API 接口、数据同步**。核心是先建数据资产目录让业务方看得见有什么数据，然后根据场景选对方式。

### 🖼️ 一个比喻：餐厅的三套供餐模式

把大数据平台想象成**中央厨房**，业务部门是**前厅客人**，数据资产是**做好的菜**。

**传统模式的痛**：
没有菜单（无数据资产目录）→ 客人不知道厨房有什么菜 → 跟服务员口述需求 → 服务员转达给厨师长 → 厨师长派单 → 厨师现做 → 端回来。一个需求走十几二十天，菜上来可能凉了（数据时效过了）。

**改进后（有数据资产目录）**：
先把菜单贴出来（数据资产目录）→ 客人看菜单就知道有什么菜 → 想吃哪个直接下单（申请）→ 审核通过 → 按菜的类型选上菜方式：

| 上菜方式 | 对应技术 | 适用场景 | 类比 |
|---|---|---|---|
| 客人自己进厨房端 | JDBC 直连 | 内部小范围，不讲究安全 | 后厨重地随便进，简单但容易出乱子 |
| 服务员端上来 | API 接口 | 数据量小，要求安全 | 服务员一碟一碟上，安全但大桌菜上不过来 |
| 提前备好送进包间 | 数据同步 | 数据量大，定期供应 | 每天固定时间把整桌菜推进包间，量够效率高 |

**数据同步的三种递送模式**：

| 模式 | 类比 |
|---|---|
| 库→库 | 中央厨房直接送到客人家厨房（两家门对门，简单但私密性差） |
| 库→文件→需求方 | 中央厨房→小区门口外卖柜→客人自取（安全隔离，主流做法） |
| 文件→文件 | 中央厨房→渡口A →摆渡船→渡口B →客人家（跨河跨网，适合集团和子公司之间） |

**API 为啥不能直接接 Hive？**
Hive 就像一口大缸——菜都在里面，但每次拿都要伸勺子搅半天。Presto 就是在缸口加了个**分菜勺**，快速定位舀一勺就走。

### 企业 vs 政务数据服务

| 维度 | 政务 | 企业 |
|---|---|---|
| 目录复杂度 | 繁琐——国标编目、资源挂载、上下级级联 | 相对简单——按主题库/专题库组织，直接开放 |
| 核心平台 | 共享交换平台（四板块：目录+交换+共享+开放） | 数据资产目录 + 同步引擎 / API 网关 |
| 审批流程 | 严格——申请→审核→交换任务 | 申请→审批→获取（流程相对短） |

### 三种获取方式对比

| 方式 | 适用场景 | 优点 | 缺点 |
|---|---|---|---|
| **JDBC 直连** | 内部小范围使用 | 最简单，直接连库 | 安全性低，暴露数据库连接信息 |
| **API 接口** | 数据量小、需要安全管控 | 不暴露后台存储，更安全 | 大数据量时性能差 |
| **数据同步** | 数据量大、周期性更新 | 适合大批量、定期推送 | 需要额外的同步引擎和调度 |

### 传统模式的痛点

很多企业还是老路子：业务部门不知道数据部门有什么数据（数据资产对业务方是黑盒）→ 提需求给数据管理员 → 数据管理员拆解分派给 IT 开发 → 做完返回给业务。**周期长、难复用、效率低。**

### 改进方案：数据资产目录

不管用 API 还是同步，首先要让业务方**看得见数据**。数据资产目录就是把治理后的成果展示出来——分两类资产：

- **API 类资产**：通过配置把库表数据生成 API（不管底层是关系库、Hive 还是 HBase），挂到目录下
- **库表类资产**：直接的物理表，挂到目录下

业务人员在目录里搜索 → 申请 → 审批通过 → 按指定方式（API 或同步）获取数据。

### 数据同步的三种模式

| 模式 | 做法 | 适用场景 |
|---|---|---|
| **库→库** | 大数据平台库直接同步到需求方库 | 上层应用少、非跨部门，简单但安全性低 |
| **库→文件→需求方** | 数据导出为文件放 FTP 服务器，需求方定时取 | 常见——运营商项目多用这种，安全隔离好 |
| **文件→文件** | 通过两个 FTP 服务器跨网桥接 | 跨网段（集团↔子公司、子公司↔子公司），两个 FTP 服务器跨双网接驳 |

### 同步任务配置流程

```
创建数据源（源端 → 目标端）
    ↓
选择要同步的表（源表 → 目标表）
    ↓
配置字段映射
    ↓
执行测试（验证数据能过去）
    ↓
挂定时调度（按月/周/天/时）
    ↓
任务监控 + 异常报警
```

> 和 [[数据治理知识体系#二、数据接入（搬进来）|数据接入]] 正好是反向的——接入是从业务库拿到大数据平台，数据同步是从大数据平台推送回业务方。

### API 方式特别提醒

大数据平台的数据资产一般存在 **Hive** 里。直接在 Hive 上生成 API → 访问效率极慢。标准做法是在 Hive 上面加一层 **Presto**（内存分布式查询引擎），通过 Presto 来读取 Hive 数据并对外提供 API，访问效率高很多。

### 和知识体系的关系

- [[数据治理知识体系#四、数据仓库（分类存放，记历史账）|数据仓库]] 里构建的 DM 层主题库/专题库 → 就是数据资产目录的内容来源
- [[数据治理知识体系#二、数据接入（搬进来）|数据接入]] 是"搬进来"→ 数据同步是"推出去"，技术手段一样，方向相反
