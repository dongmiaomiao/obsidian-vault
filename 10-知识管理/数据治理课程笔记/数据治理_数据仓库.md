# 数据治理 · 数据仓库

## ✨ 这是啥

> **数据仓库不是某个具体产品，而是一整套数据管理的方法论和建设过程——把不同来源的数据经过抽取、清洗、转换后，统一存到面向主题的分类"仓库"里，记录历史变化，支撑数据分析。**

数据库里存的是**当前状态**（一条记录覆盖旧记录），数据仓库里存的是**历史变化**（每条记录带时间戳，只追加不删除）。

---

## 🎯 有啥用

### 没有数据仓库会怎样？（以电信业务为例）

| 对比维度 | 没有数仓（直接查业务库） | 有数据仓库 |
|---|---|---|
| **查询速度** | 从海量详单表做多表关联，慢 | 从 DW 层直接取汇总指标，快 |
| **稳定性** | 上游业务字段/逻辑变化，下游立刻跟着改 | 数仓做了清洗转换，上游变动不影响下游 |
| **历史数据** | 业务系统只存当前数据，查不到去年的指标 | 数仓记录历史变化，随时回溯 |
| **数据质量** | 源端质量参差不齐 | 做了清洗和标准化，数据质量高 |
| **指标支撑** | 临时写 SQL，改一次变一次 | 稳定支撑几百上千个指标 |

> **结论**：要做数据分析、数字化转型，数据仓库是必不可少的。它是数据治理落地的"容器"和"骨架"。

---

## 🛠️ 怎么用

### 一、数据库 vs 数据仓库（核心区分）

| 维度 | 数据库（Database） | 数据仓库（Data Warehouse） |
|---|---|---|
| **定位** | 具体产品（Oracle / MySQL / 达梦） | 方法论 + 建设过程 |
| **数据特征** | 当前时点数据，记录会被覆盖 | 历史变化数据，只追加不覆盖 |
| **用途** | 支撑业务交易（OLTP） | 支撑数据分析（OLAP） |
| **数据范围** | 单一业务系统 | 集成多系统多类型数据 |
| **组织方式** | 按业务流程组织 | 按主题分类组织 |
| **主键/时间戳** | 有主键，无时间戳 | 无主键，有时间戳（通过时间戳定位历史） |

### 二、数据仓库的四大特性

| 特性 | 解释 | 类比（新发地市场） |
|---|---|---|
| **① 面向主题** | 按主题分类存储，如订单主题、客户主题、产品主题 | 蔬菜大厅、牛羊肉大厅、水果大厅——要什么去对应大厅找 |
| **② 集成** | 把不同来源的数据汇总到一个统一的仓库里 | 全国各地通过铁路、飞机、汽车把农产品运到新发地 |
| **③ 相对稳定** | 数据入库后不修改，只用于分析 | 货物进了大厅，只是展示和取用，不会去改它 |
| **④ 反映历史变化** | 每条数据带时间戳，保留历史快照 | 地窖里记录了每天摘下来的菜，能看出哪天摘了什么 |

### 三、数据仓库建设概览

数仓建设不只是"建一个库"，而是一整套流程：

```
业务系统 → 抽取(Extract) → 转换清洗(Transform) → 加载(Load) → 数据仓库
                                                              ↓
                                                     ODS → DW → DM → 报表/指标
```

其中包括：
- 数据抽取工具（参考 [[数据治理_数据接入]]）
- 分层设计（ODS 贴源层 → DW 汇总层 → DM 数据集市层）
- 主题划分
- 数据质量管理（参考 [[数据治理_数据质量]]）
- 元数据管理（参考 [[数据治理_元数据]]）
