# 数据治理 · 数据接入

## ✨ 这是啥

> **数据接入是把各种不同类型的数据源（数据库、API、文件、实时流等）汇总到一个统一的数据资源池（大数据平台），为后续的数据治理和分析做准备。**

数据接入是数据治理的基础性工作。狭义的数据治理只包含标准/元数据/质量/安全四个范围，广义的数据治理把**接入→治理→资产→服务**全链路都包含了进来。

---

## 🎯 有啥用

- 把分散在不同业务系统的数据集中管理
- 为数据治理（清洗、标准化）提供原材料
- 为上层数据分析和应用提供统一的数据基础

---

## 🛠️ 怎么用（核心知识）

### 一、数据类型分类

| 类型 | 典型代表 | 特征 |
|---|---|---|
| **结构化数据** | 关系数据库（MySQL/Oracle） | 二维表存储，数据治理的主流场景 |
| **半结构化数据** | Kafka / MongoDB / ES / HBase | JSON 串、对象存储 |
| **非结构化数据** | 图片、文件、音频、视频 | 格式不固定 |
| **实时数据** | CDC 监听（Flink CDC）+ 传感器/事件消息 | 日志变化捕获或消息队列触发 |

---

### 二、三种接入方式

| 方式 | 适用场景 | 特点 |
|---|---|---|
| **整库接入** | 集群初始化，一次性大批量接历史数据 | 批量快，可过滤不需要的表 |
| **单表接入** | 日常增量数据，每天针对每张表单独接 | 灵活，按需配置每张表的策略 |
| **SQL 接入** | 特定筛选条件的数据 | 灵活但用得少 |

**典型流程**：新集群搭建 → 用**整库接入**做初始化把历史数据全接进来 → 之后每天用**单表接入**处理新增数据。

---

### 三、四种数据策略

| 策略 | 做法 | 适用场景 |
|---|---|---|
| **全量·覆盖** | 每天全量拉取，覆盖前一天的数据 | 代码表/编码表（变化小、不存历史） |
| **全量·追加** | 每天全量拉取，带时间戳追加存储 | 核心业务表（需要保存每天的快照，有数据冗余） |
| **增量·追加** | 只拉取新增的数据，追加到历史数据后 | 日志表/流水表（历史不会变化） |
| **实时（CDC）** | 监听数据库日志变化，捕获 insert/update/delete | 对时效要求高的场景 |

> **区分关键**：是否要存历史变化 → 要存 → 追加模式；不存历史 → 覆盖模式；历史不会变 → 纯增量。

---

### 四、常用技术工具对比

| 工具 | 类型 | 现状 | 适用规模 |
|---|---|---|---|
| **Sqoop** | Hadoop 开源组件 | ⭐ 最常用、最稳定 | 大批量接入 OK，小表反而慢 |
| **DataX** | 阿里开源 | 社区不活跃，bug 多，用得越来越少 | 单机版，大并发场景不适用 |
| **Kettle** | 开源 ETL | 曾经火，现在少了 | 小数据量，大场景稳定性差 |
| **Informatica** | 商用 | 稳定但贵 | 企业级 |
| **Data Stage** | IBM 商用 | 传统工具 | 企业级 |

**Sqoop 接入流程**（典型四步）：
1. 获取源端元数据信息（表结构、字段）
2. 转换数据类型，在 Hive 上建表并指定 HDFS 路径
3. 从 MySQL 抽数据到临时目录（转为文件）
4. Load 到目标 HDFS 路径 → 接入完成
