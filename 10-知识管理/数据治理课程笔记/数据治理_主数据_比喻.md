# 数据治理 · 主数据 比喻两则

---

## 比喻一：主数据管理系统 = 城市交通指挥中心

### 场景设定

一个大城市，每个路口都有红绿灯（各个业务系统），每个路口的交通流量都不同。

### 没有交通指挥中心（没有主数据管理）

每个路口各自为政——A 路口的红绿灯自己判断绿灯多久，B 路口也自己判断，C 路口也自己判断。结果：

- A 路口放了一批车过去，但 B 路口还没准备好接收，路上全堵了
- 一个方向的路在修（相当于一个系统的客户信息变了），其他路口不知道，车还往那开
- 结果是**四个方向全堵死**——数据不一致，各自为政

### 有了交通指挥中心（建了主数据管理系统）

所有路口把实时车流量数据**双向**传给指挥中心：

```
各路口车流量 → 指挥中心大屏 → 统一计算红绿灯时长 → 同步下发到各路口
```

| 流程 | 交通版 | 数据版 |
|---|---|---|
| 收集信息 | 每个路口上报实时车流量 | 业务系统发生变更 → 通知 MDM |
| 统一处理 | 指挥中心根据全城流量统一算绿灯时长 | MDM 统一校验、清洗 |
| 同步下发 | 指挥中心把新的信号时长下发给所有受影响的路口 | MDM 把变更同步到所有相关系统 |
| 实时性 | 晚一秒就是事故（救护车被堵） | 晚一秒就是事故（库存超卖） |

**核心心法**：指挥中心不是"记录历史流量"的（那是数据仓库的事），而是**实时调度交通**的——哪个路口变了，立刻通知其他路口调整。

## 比喻二：主数据 vs 数据仓库 = 身份证系统 vs 人口普查

### 场景设定

中国政府有两套跟人口相关的系统。

### 身份证系统（主数据管理）

- 你出生登记、结婚、搬家、换工作——**任何一个变更都要及时更新**
- 公安系统改了你的户籍信息，**实时同步**到民政、社保、税务、银行等所有需要用你身份信息的系统
- 你在公安局改个名字，第二天去银行办业务，银行系统已经看到新名字了
- **双向、高实时性、只管理核心身份信息**

### 人口普查（数据仓库）

- 每十年做一次，统计全国人口的年龄结构、教育水平、就业分布
- 从各个系统（公安、民政、社保）**单向**收集数据，统一汇总分析
- 分析结果用于国家决策——不反馈回各个业务系统
- **单向、T+1 甚至 T+年、管理全面数据**

### 对比一张表

| 维度 | 身份证系统（主数据） | 人口普查（数据仓库） |
|---|---|---|
| 数据流向 | 双向：你改了 → 同步到所有系统 | 单向：收集 → 汇总分析 |
| 实时性 | 改名字当天全网生效 | 十年做一次 |
| 数据范围 | 只有核心身份信息 | 年龄、学历、就业、住房…全量 |
| 目的 | 保证每个系统用的都是同一个你 | 了解国家人口状况做决策 |

> **两者也可以配合**：身份证系统的核心数据，可以直接作为人口普查的一个高质量输入源。反过来，人口普查的统计结果，也可以辅助身份证系统的管理决策。
